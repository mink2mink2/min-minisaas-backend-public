# R/20 — 인수 기준 (Acceptance Criteria)

> 각 유저 스토리에 대한 구체적인 합격/불합격 기준이다.
> 테스트 케이스 작성 시 이 문서를 기준으로 한다.

---

## AUTH 도메인

### AC-A01: Google 소셜 로그인

| # | 조건 | 기대 결과 |
|---|------|----------|
| 1 | 유효한 Firebase ID Token 전송 | HTTP 200, `access_token`(JWT) 및 사용자 정보 반환 |
| 2 | 만료된 Firebase ID Token 전송 | HTTP 401, `{"detail": "Invalid or expired token"}` |
| 3 | 형식이 잘못된 토큰 전송 | HTTP 401 |
| 4 | 신규 사용자 최초 로그인 | DB에 사용자 레코드 생성 후 JWT 반환 |
| 5 | 기존 사용자 재로그인 | 기존 레코드 반환 (중복 생성 없음) |
| 6 | `provider` 필드가 응답에 `"google"` 포함 | ✅ 검증 |

## CHAT 도메인

### AC-C01: 1:1 채팅방 생성

| # | 조건 | 기대 결과 |
|---|------|----------|
| 1 | 새 상대방과 채팅방 생성 | HTTP 201, 새 채팅방 ID 반환 |
| 2 | 이미 존재하는 채팅방 대상 재요청 | HTTP 200, 기존 채팅방 반환 (중복 생성 없음) |
| 3 | 자기 자신과 채팅방 생성 시도 | HTTP 400, `{"detail": "Cannot create chat with yourself"}` |
| 4 | 존재하지 않는 사용자 ID | HTTP 404 |

### AC-C02: 채팅방 목록 조회

| # | 조건 | 기대 결과 |
|---|------|----------|
| 1 | 내가 참여한 채팅방 있음 | HTTP 200, 최신 메시지 순 정렬 목록 |
| 2 | 채팅방 없음 | HTTP 200, 빈 배열 `[]` |
| 3 | 다른 사용자의 채팅방은 포함 안 됨 | ✅ 격리 확인 |

### AC-C03: 실시간 메시지 (WebSocket)

| # | 조건 | 기대 결과 |
|---|------|----------|
| 1 | 유효한 JWT로 WS 연결 | 연결 수립 성공 |
| 2 | 토큰 없이 WS 연결 | 연결 거부 (4001 코드) |
| 3 | 메시지 전송 | 동일 채팅방 다른 참여자에게 실시간 전달 |
| 4 | 메시지 전송 시 DB 저장 | 이력 조회로 확인 가능 |
| 5 | 연결 종료 후 재연결 | 이전 메시지 이력 유지 |

### AC-C04: 메시지 이력 조회

| # | 조건 | 기대 결과 |
|---|------|----------|
| 1 | 채팅방 참여자가 요청 | HTTP 200, 시간 역순 메시지 목록, 페이지네이션 |
| 2 | 채팅방 비참여자가 요청 | HTTP 403 |
| 3 | `limit`, `offset` 파라미터 | 올바른 페이지네이션 적용 |

## 공통 인수 기준

### 인증/권한

| 조건 | 기대 결과 |
|------|----------|
| `Authorization: Bearer {token}` 헤더 누락 | HTTP 401 |
| 만료된 JWT | HTTP 401, `{"detail": "Token has expired"}` |
| 위조된 JWT | HTTP 401 |
| 권한 없는 리소스 접근 | HTTP 403 |

### 입력 유효성 검사

| 조건 | 기대 결과 |
|------|----------|
| 필수 필드 누락 | HTTP 422, 필드별 에러 메시지 |
| 필드 타입 오류 | HTTP 422 |
| 문자열 최대 길이 초과 | HTTP 422 |

### Rate Limiting

| 조건 | 기대 결과 |
|------|----------|
| 공개 엔드포인트 분당 60회 초과 | HTTP 429, `Retry-After` 헤더 포함 |
| 게시글 작성 분당 10회 초과 | HTTP 429 |
| 댓글 작성 초당 1회 초과 | HTTP 429 |

### DB 스키마 안전성

| # | 조건 | 기대 결과 |
|---|------|----------|
| 1 | Alembic metadata 구성 시 운영 DB에 이미 존재하는 도메인 모델이 registry에서 누락되지 않음 | `alembic check`/`autogenerate`가 기존 테이블을 삭제 대상으로 오인하지 않음 |
| 2 | `BaseModel` 공통 컬럼(`updated_at`, `is_deleted`)을 기대하는 모델이 운영/개발 DB와 불일치 | 차이는 수동 Alembic migration으로 보정되고 `upgrade head`가 재실행 가능하게 동작 |
